## Code Review — PR #46: Sistema de Avaliações (RF010)

### 🔴 CRITICAL

**C1 — URL prefix mismatch: frontend ↔ backend (404 em todas as chamadas)**

O frontend (`providers.ts`) chama `/api/providers/${id}/reviews` (plural), mas o backend monta o router `provider.ts` em `/api/provider` (singular, linha 31 do server.ts). As reviews NUNCA vão chegar ao endpoint — 100% de 404.

```ts
// frontend/src/services/providers.ts
await api.get(`/api/providers/${providerId}/reviews`);  // ❌ 404

// backend/src/server.ts
app.use('/api/provider', providerWizardRoutes);   // ← singular!
// backend/src/routes/provider.ts
router.post('/:userId/reviews', ...);  // monta em /api/provider/:userId/reviews
```

**Solução:** Ou mover os endpoints de review para `providers.ts` (plural), ou corrigir a URL no frontend para `/api/provider/${providerId}/reviews`.

---

**C2 — GET reviews exige autenticação (bloqueia visitantes)**

`provider.ts` linha 13: `router.use(requireAuth)` aplica-se a TODAS as rotas do router, incluindo `GET /:userId/reviews`. Visitantes não autenticados na página pública `/prestador/:id` recebem 401 ao tentar carregar as avaliações.

```ts
const router = Router();
router.use(requireAuth);  // ← afeta todas as rotas abaixo

// ...mais rotas...

router.get('/:userId/reviews', (req, res) => {  // ❌ requer auth
  // Deveria ser público — qualquer um pode ver avaliações
});
```

**Solução:** Mover `requireAuth` para as rotas específicas que precisam (wizard, portfolio upload) em vez de aplicar globalmente. Ou criar um sub-router sem auth para os endpoints de review.

---

**C3 — Race condition: verificação de duplicata + INSERT sem transação**

Dois requests concorrentes podem ambos passar em `getClientReviewForContact()` antes de qualquer INSERT, criando reviews duplicadas. Além disso, o INSERT + UPDATE (avg rating) não está em transação — se o UPDATE falhar, a média fica inconsistente.

```ts
// Sem transação, sem UNIQUE constraint no DB
const existing = getClientReviewForContact(clientId, contactId);
if (existing) { ... }  // ← race: ambos veem null
const review = createReview(...);  // ← ambos inserem
```

**Solução:** Adicionar `UNIQUE(client_id, contact_id)` na tabela reviews + wrap em `db.transaction(() => { ... })`.

### ⚠️ WARNINGS

**W1 — `hasClientContactedProvider` recebe `user_id` como `providerId`**

A função espera semanticamente um `providerId` mas a rota passa `profile.user_id` (UUID do usuário). Funciona porque `contact_history.provider_id` armazena `users.id` (linha 202 do db.ts), mas o nome engana — alguém no futuro pode passar `profile.id` (provider_profiles.id) e a verificação sempre retornará false.

**Solução:** Renomear o parâmetro para `providerUserId` ou ajustar a query para aceitar ambos os formatos.

**W2 — Frontend não envia `contactId` no submit**

O backend aceita `contactId` para linkar a review ao contato, mas o `ReviewSection.tsx` nunca envia esse campo. A regra de negócio "uma review por contato" fica inaplicável via UI.
