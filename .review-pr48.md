## Code Review — PR #48: Denúncias + Avaliações (RF010, RF011)

> ⚠️ **Nota sobre overlap:** Este PR inclui o código do PR #46 (reviews) + denúncias. Os problemas C1–C3 de reviews se aplicam igualmente aqui. Se este PR for mergeado primeiro, o PR #46 pode ser fechado como duplicado.

### 🔴 CRITICAL

**C1 — URL prefix mismatch: frontend ↔ backend (reviews + reports dão 404)**

O frontend (`providers.ts`) chama `/api/providers/${id}/reviews` e `/api/providers/${id}/report` (plural), mas o backend monta o router `provider.ts` em `/api/provider` (singular, linha 31 do server.ts).

```ts
// frontend:  /api/providers/...  (plural, linha 32 do server.ts)
// backend:   /api/provider/...   (singular, linha 31 do server.ts)
await api.get(`/api/providers/${id}/reviews`);   // ❌ 404
await api.post(`/api/providers/${id}/report`);   // ❌ 404
```

**Solução:** Mover os endpoints para `providers.ts` (plural) ou corrigir as URLs no frontend.

---

**C2 — GET reviews exige autenticação (bloqueia visitantes)**

`provider.ts` linha 13: `router.use(requireAuth)` global. `GET /:userId/reviews` fica inacessível para visitantes não logados.

**Solução:** Aplicar `requireAuth` seletivamente nas rotas que precisam, não globalmente.

---

**C3 — Race condition: verificação de duplicata sem transação**

`getClientReviewForContact()` + `createReview()` sem transação — dois requests concorrentes podem criar reviews duplicadas. Também não há `UNIQUE(client_id, contact_id)` no banco.

---

### ⚠️ WARNINGS

**W1 — `hasRecentReport` concatenação SQL frágil**

```ts
AND created_at > datetime('now', ? || ' hours')
// parâmetro: `-24` → vira `datetime('now', '-24 hours')`
```

Funciona com SQLite, mas é frágil. Se alguém passar `24` sem o sinal negativo, a query retorna registros futuros. Melhor computar a data de corte em JS:

```ts
const cutoff = new Date(Date.now() - hours * 3600 * 1000).toISOString();
// ... WHERE created_at > ?
```

---

**W2 — `getPendingReports()` existe mas sem rota de admin**

A função retorna denúncias pendentes, mas não há endpoint REST para o painel admin consumir. As denúncias ficam acumuladas no banco sem mecanismo de revisão.

**Solução:** Adicionar `GET /api/admin/reports` (protegido por `requireAuth` + `requireRole('admin')`).

---

**W3 — ReportModal não reseta estado ao reabrir**

Se o usuário abre o modal, fecha, e reabre — o `reason` e `desc` anteriores permanecem. O estado deveria resetar quando `isOpen` transiciona para `true`.

```ts
// Sugestão: resetar no useEffect quando isOpen muda
useEffect(() => { if (isOpen) { setReason(REASONS[0]); setDesc(''); setError(''); } }, [isOpen]);
```

---

### 💡 SUGGESTIONS

- **S1 — Separar reviews e reports em services distintos.** `providers.ts` está acumulando responsabilidades (perfil, portfolio, reviews, reports). Um `reviews.ts` e `reports.ts` melhorariam a organização.
- **S2 — Adicionar `UNIQUE(reporter_id, reported_provider_id)` com constraint de tempo?** A regra de 24h via `hasRecentReport` é boa, mas não impede spam de longa duração. Um `UNIQUE` com `created_at` window seria ideal mas complexo em SQLite. OK para MVP.
- **S3 — `InteractiveStars` minificado demais.** O código de 13 linhas é funcional mas sacrifica legibilidade. A versão do PR #46 (34 linhas, expandida) é mais manutenível.
